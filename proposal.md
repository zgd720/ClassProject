## 1. 项目概述

### 1.1 项目背景

高校数字化建设积累了多源异构的学生行为数据（成绩、消费、借阅、上网、门禁等），但缺乏直观、智能的分析工具。本项目构建一个轻量级学业行为分析系统，帮助辅导员快速识别学生的优势与风险，并为学生提供个性化学习建议。

### 1.2 项目目标

- 构建学生五维画像（学业、借阅、消费、上网、作息）。
- 实现多因子加权风险评分模型，替代简单的挂科数判定。
- 动态生成行为标签（规则标签 + 班级相对标签）。
- 集成大模型 API，基于全维度数据生成个性化学习建议。
- 提供班级群体分析看板，支持交叉分析与异常学生关注。
- 支持 Docker Compose 一键部署。

### 1.3 用户角色

| 角色 | 说明 | 主要操作 |
| :--- | :--- | :--- |
| 辅导员 | 查看所带班级学生数据 | 导入数据、查看学生画像、群体分析看板、生成 AI 建议 |
| 学生 | 查看个人画像和建议 | 登录、查看本人五维画像及 AI 建议 |
| 管理员 | 系统初始化与用户管理（可选） | 用户导入、角色分配 |

## 2. 核心业务流程

1. 辅导员通过 CSV 文件导入学生信息、成绩、消费、借阅、上网行为、门禁记录（共 6 类 CSV）。
2. 系统清洗并存储数据，自动计算每个学生的五维得分（学业/借阅/消费/上网/作息），综合评定风险等级，并生成行为标签。
3. 辅导员查看学生列表，按风险等级或行为标签筛选，点击学生进入综合画像页。
4. 学生画像页展示五维雷达图（叠加班级均值）、行为标签、各维度趋势面板。
5. 辅导员点击"生成 AI 建议"，系统调用大模型 API 并基于全维度数据返回个性化建议（含优劣势分析）。
6. 班级看板提供风险概览、五维均值对比、异常关注列表、群体共性分析、交叉散点图。

## 3. 功能需求

### 3.1 模块结构

| 模块 | 子功能 | 说明 |
| :--- | :--- | :--- |
| 数据管理模块 | CSV 导入（6 类）、数据预览、数据清空 | 辅导员导入全部数据源 |
| 学生画像模块 | 增强列表、五维雷达图、行为标签、各维度趋势面板 | 含班级均值叠加对比 |
| 风险评分模块 | 五维加权评分、行为标签生成、多条件筛选 | 替代简单挂科数判定 |
| AI 建议模块 | 全维度 Prompt、优劣势分析、调用限制、降级兜底 | 每学生每天限 3 次 |
| 班级看板模块 | 风险概览、均值对比、异常关注、共性分析、交叉散点 | 5 区联动 |
| 用户认证模块 | JWT 登录、路由守卫、角色权限 | 不变 |

### 3.2 数据管理模块

- **CSV 导入**：支持上传 6 类 CSV 文件：
  1. 学生基本信息表
  2. 成绩表
  3. 消费记录表
  4. 借阅记录表
  5. 上网行为表（每日记录：日均时长、学习/娱乐/社交占比）
  6. 门禁记录表（刷卡时间、进出方向）
- **数据预览**：展示最近导入的若干条记录，供用户确认。
- **数据清空**：一键清空所有业务表（保留用户表）。

### 3.4 风险评分模块（增强）

#### 3.4.1 五维加权评分模型

每维 0-20 分，总分 100 分（5 维 × 20 分）。分数越低风险越高。

| 维度 | 数据来源 | 低风险(15-20 分) | 中风险(8-14 分) | 高风险(0-7 分) |
| :--- | :--- | :--- | :--- | :--- |
| 学业水平 | course_score | GPA ≥ 3.0，挂科 = 0 | GPA 2.0-3.0，挂科 = 1 | GPA < 2.0，挂科 ≥ 2 |
| 借阅投入 | library_borrow | 本学期借阅 ≥ 5 本 | 2-4 本 | ≤ 1 本 |
| 消费健康 | consumption | 日均 30-60 元，三餐规律 | 日均 < 30 或 > 80，略不规律 | 日均 < 15 或 > 120，极不规律 |
| 上网健康 | internet_behavior | 日均 < 3h，学习占比 > 40% | 3-6h，学习占比 20-40% | > 6h 或娱乐占比 > 70% |
| 作息规律 | dormitory_access | 晚归频率 < 10%，出入规律 | 晚归 10%-30% | 晚归 > 30%，或凌晨频繁出入 |

**综合风险等级**：

| 总分 | 风险等级 | 颜色 |
| :--- | :--- | :--- |
| < 40 | 高风险 | 🔴 红 |
| 40-65 | 中风险 | 🟡 黄 |
| > 65 | 低风险 | 🟢 绿 |

#### 3.4.2 行为标签

标签在查询时动态计算，不存储固定值。

**规则标签**（绝对值触发）：

| 标签 | 触发条件 |
| :--- | :--- |
| 🌙 夜猫子 | 近 30 天晚归频率 > 30% |
| 📱 网络沉迷风险 | 日均上网 > 6h 且娱乐占比 > 60% |
| 📚 图书馆常客 | 本学期借阅 ≥ 10 本 |
| 🍜 高消费型 | 日均消费 > 100 元 |
| 💰 低消费型 | 日均消费 < 15 元 |
| ⚠ 作息紊乱 | 连续 3 天归寝时间标准差 > 3h |
| 📖 学习型上网 | 上网学习占比 > 50% |

**相对标签**（班级排名触发）：

| 标签 | 触发条件 |
| :--- | :--- |
| 🔺 上网时长班级前 20% | 日均上网时长位于班级前 20% |
| 🔻 借阅量班级后 20% | 本学期借阅量位于班级后 20% |
| 🔺 消费金额班级前 20% | 日均消费金额位于班级前 20% |
| 🔻 学业排名班级后 20% | GPA 位于班级后 20% |
| 🔺 学业排名班级前 20% | GPA 位于班级前 20% |

### 3.5 AI 建议模块（增强）

#### 3.5.1 触发方式

学生详情页提供"生成 AI 建议"按钮。后端收集该生五维全部数据，构造 Prompt 调用大模型 API。

#### 3.5.2 Prompt 设计

```
你是一名经验丰富的大学辅导员。请根据以下学生多维度数据，给出个性化学习建议：

【学业】平均绩点：{gpa}，挂科数：{failCount}，各学期绩点趋势：{gpaTrend}
【借阅】本学期借阅：{borrowCount}本，偏好类别：{favCategory}
【消费】日均消费：{avgConsume}元，主要消费类型：{consumeType}，三餐规律度：{mealRegularity}
【上网】日均上网：{dailyInternet}h，学习占比：{studyRatio}%，娱乐占比：{entertainmentRatio}%
【作息】近30天晚归次数：{lateReturn}，归寝时间规律度：{scheduleRegularity}

该生当前风险等级：{riskLevel}，行为标签：{tags}

要求：
1. 结合以上所有维度的数据，给出针对性建议
2. 先指出该生最突出的优势（1-2条），再指出最需改善的方面（1-2条）
3. 建议具体可操作，避免空洞说教
4. 不超过300字
```

#### 3.5.3 降级与限制

- **超时处理**：API 调用 5 秒超时，返回兜底文案。
- **错误降级**：网络错误或非 200 状态码，返回兜底文案。
- **兜底文案**："当前无法获取 AI 建议，请稍后再试。建议您多与任课老师沟通，并合理安排学习时间。"
- **调用限制**：每学生每天最多生成 3 次 AI 建议，超限后返回提示。

#### 3.5.4 文档要求

`docs/ai_module.md` 需包含：
- 使用的模型名称、API 提供方
- 请求/响应示例
- Prompt 设计思路及截图
- 调用流程图
- 成本控制与限流说明

### 3.7 用户认证模块

- 使用 JWT 实现登录认证。
- 前端路由守卫：未登录状态下无法访问除登录页外的任何页面。
- 权限控制：学生只能查看本人数据；辅导员可查看所带班级全部学生数据（班级字段关联）。
- 密码使用 BCrypt 加密。

## 4. 数据需求

### 4.1 数据表设计（8 张）

| 表名 | 主要字段 | 说明 |
| :--- | :--- | :--- |
| `user` | id, username, password(BCrypt), role, name, student_id(可为空) | 用户表 |
| `student` | id, student_id, name, gender, major, class_name, enrollment_year | 学生基本信息 |
| `course_score` | id, student_id, semester, course_name, score, gpa | 成绩表 |
| `library_borrow` | id, student_id, borrow_date, book_name, category | 借阅记录 |
| `consumption` | id, student_id, consume_time, amount, type | 消费记录 |
| `internet_behavior` | id, student_id, record_date, daily_duration, study_ratio, entertainment_ratio, social_ratio | 上网行为每日记录 |
| `dormitory_access` | id, student_id, access_time, direction(in/out) | 门禁刷卡记录 |
| `risk_log` | id, student_id, risk_level, total_score, dimension_scores(JSON), behavior_tags, update_time | 风险记录（含各维度明细） |

> 所有业务表均包含 `create_time`, `update_time`, `deleted`（逻辑删除）字段。

### 4.2 数据来源

- 使用自行生成的模拟数据（CSV 格式），共 6 类 CSV 文件。
- 数据量要求：学生数 ≥ 20 人，每人至少 3 门课成绩，≥ 10 条借阅记录，≥ 20 条消费记录，≥ 30 天上网行为记录，≥ 30 天门禁记录。

### 4.3 数据生成脚本需求

需编写 Python 脚本 `scripts/generate_data.py`，生成 6 类 CSV 模拟数据。生成规则需体现：
- 学生间的合理差异（存在高风险/中风险/低风险学生）
- 行为维度与学业维度的弱关联（如高上网娱乐占比倾向于低 GPA）
- 数据内部一致性（同一学生不同维度数据关联合理）

## 5. 非功能需求

| 类别 | 具体要求 |
| :--- | :--- |
| 性能 | 单次 API 请求 ≤ 10 秒；列表查询响应 ≤ 2 秒；看板加载 ≤ 5 秒；标签/评分计算不拖慢列表查询 |
| 可用性 | 前端提供 Loading 状态；所有操作均有成功/失败提示；雷达图/散点图有数据为空的占位提示 |
| 可靠性 | 后端全局异常处理（@RestControllerAdvice）；AI 模块降级不导致主流程崩溃；参数校验使用 @Valid |
| 可维护性 | 遵循三层架构（Controller-Service-Mapper）；Swagger 文档可访问；关键计算逻辑有注释 |
| 安全性 | 密码 BCrypt 加密；JWT 过期时间 ≥ 1 天；CSRF 忽略（本地开发） |

## 8. 验收标准

- `docker compose up --build` 一键启动，通过 `http://localhost` 访问前端，通过 `http://localhost:8080/swagger-ui/index.html` 访问 API 文档。
- 辅导员可成功导入 6 类 CSV 文件，学生列表正确显示五维迷你进度条和风险标签。
